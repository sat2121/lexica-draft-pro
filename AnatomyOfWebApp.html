Core Philosophy

1

Single HTML File

The entire application is self-contained. This maximizes portability and simplifies deployment, requiring no build tools or complex server setup.

All data, including user profiles and drafts, is stored client-side in the browser's `localStorage`, ensuring user privacy and a fast, offline-capable experience.

Application Initialization Flow

1. DOMContentLoaded

The browser finishes loading the HTML file.

2. init()

The main script function is called.

3. loadState()

Drafts and user profile are loaded from `localStorage` into the `state` object.

4. Render & Populate

Functions like `renderTemplateSelector` and `updateUIWithProfileData` build the UI from data.

5. App Ready

The dashboard is displayed, and the application is idle, awaiting user input.

The Data Engine

templatesDB

Defines contract recipes

defaultClauses: ['id1', 'id2']
‚Üí

clausesDB

Contains dynamic clause content

'id1': { text: (data) => `...` }

Templates provide a list of clause IDs, which are used as keys to look up the corresponding dynamic text functions in the `clausesDB`. This modular system allows any contract to be built from a universal set of clauses.

Future Roadmap

The current architecture provides a solid foundation for a single-user tool. The next evolution involves migrating from `localStorage` to a real-time database like Firebase Firestore to enable multi-user collaboration, commenting, and version history.
